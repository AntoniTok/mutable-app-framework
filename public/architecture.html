Mutable App Framework — Architecture

A self-modifying app framework on Cloudflare Workers. An app's source code lives as data in a Durable Object; each request runs the live version in an isolated Dynamic Worker; edits — manual or by the AI assistant — create new versions with no redeploy.

01System overview

One Worker (src/server.ts) is the entry point. It routes to Durable Objects (trusted, per-room state) and spawns Dynamic Workers (the untrusted app). Everything is keyed by a room id — each room is a fully isolated app instance with its own code, history, data and realtime state.

The one idea
Trusted framework code (Worker + DOs) holds all the power. The untrusted app holds none — it starts with a single handle, env.SYSTEM, and must ask the broker for every capability. New powers are added by adding broker methods, never by handing the app a raw binding.

02What actually runs the source code

The code stored as data is executed by the Cloudflare Worker Loader — the worker_loaders binding named LOADER. It creates a Dynamic Worker: a fresh V8 isolate loaded at runtime from the code strings, with no deploy. All of it lives in src/agent/runner.ts.

The load call

// runner.ts — the ONE place untrusted code runs
const worker = env.LOADER.get(workerId, async () => ({
  mainModule, modules,          // bundled code
  compatibilityDate: "2026-01-01",
  compatibilityFlags: ["nodejs_compat"],
  limits: { cpuMs, subRequests },  // #5 guardrails
  env: { SYSTEM: broker },      // the ONLY binding
  globalOutbound: null,        // no network (#12)
  tails: [DynamicWorkerTail]    // logs/exceptions
}));

const entry = worker.getEntrypoint();  // app default export
const res = await entry.fetch(request); // ← app runs here

How it behaves

  • Isolate cache key = app-<room>-<contentHash>. Same code ⇒ warm reuse; any edit ⇒ new hash ⇒ new isolate. The factory runs only on a cold cache.
  • Bundling is @cloudflare/worker-bundler (esbuild inside workerd), skipped when a precompiled build exists in the builds table (limitation #4).
  • Runs in its own isolate, separate from the host worker. It cannot import real bindings — it only sees { SYSTEM }.
  • A framework-injected adapter (__host_entry__.js) wraps the app so the author writes only plain exports (fetch, applyAction, view, seats, onSchedule, onUpgrade…). It never imports cloudflare:workers.
Who spawns it
/preview/* → the entry Worker (handlePreview) runs the app in-worker so the response can stream. The realtime coordinator (in AppHost) runs the app's pure applyAction/view. AppScheduler's alarm runs the app's onSchedule. All three go through the same runner.ts and share the same cached isolate.

03Code as data — the storage model

An app's source is plain-text rows in the AppHost DO's SQLite (src/agent/schema.ts). Nothing is mutated in place — every edit inserts a new version and a fresh set of file rows. Rollback is just moving the live pointer to a different version number.

The live pointer { activeVersion, status, templateId, lastError } lives in synced Agent state, never in SQL (invariant #3 — state is broadcast to clients, code must not be). The app's runtime data lives in separate DOs (AppData / AppSql / AppScheduler). AppHost's app_data table holds only framework scopes: __room__, __egress__, __limits__, __secrets__, __email__, __upgrade__.

04Request path — running the live app

ANY /preview/<path>?room=<id> serves the live app, streaming the response.

1
Browser → Entry Worker handlePreview; room resolved from ?room=.
2
Worker → AppHost.getRunManifest() (RPC): the live version's files + entrypoint + limits. Lazily seeds the room on first touch.
3
Worker → runner.runApp() : env.LOADER.get() gets/creates the app isolate (cold cache uses the builds cache via getBuild).
4
App runs in its Dynamic Worker; may call env.SYSTEM.request*() → capability stubs → AppData/AppSql/R2/etc. directly (never back through AppHost).
5
Response streams back through the Worker; HTML gets <base href="/preview/"> injected via HTMLRewriter (also streaming).

The HTTP data path never flows through the DO — AppHost only hands over code. Storage calls bypass AppHost entirely (limitation #3).

Inside setFiles() — the build gate

A
Insert new version + file rows (always — even a broken build is saved so it can be inspected/rolled back).
B
Bundle with esbuild (bundleApp). On failure: version saved, NOT promoted, status:"error" — the AI reads the error and self-heals.
C
On success persist the build (builds), advance activeVersion, run the app's onUpgrade data migration on a forward promote (#10).
D
Broadcast {type:"reload"} (debounced via a DO alarm) so every connected client reloads onto the new code.
Invariant
Code that doesn't compile is never promoted. The live pointer stays on the last good version. This is the whole safety net for self-modification.

06Realtime path — multiplayer without giving the app a socket

The untrusted app never holds a WebSocket. The trusted coordinator (inside AppHost) owns sockets, seats, presence, persistence and broadcast, and drives the app's pure functions in the sandbox.

1
Client WS → /agents/app-host/<room>?token=<id> — AppHost.onConnect assigns a seat.
2
Action frame → coordinator calls the app's pure applyAction(state, action, ctx) via runner.reduce() in the sandbox.
3
New state persisted in the __room__ scope; transitions are serialized (a promise chain) so frames don't race.
4
Per-viewer projection optional pure view(state, ctx) hides information (poker hands) — each client gets its own frame.
The escape hatch: requestRoom
When an update originates outside a WS action (HTTP handler, webhook, a scheduled task), the app can broadcast/send/presence via the Room capability, which reaches AppHost's live sockets by RPC and delivers {type:"app",data} frames. The app still holds no socket.

Limitation #3 — no funneling
Runtime-data DOs (AppData / AppSql / AppScheduler) are reached directly by the capability stubs (idFromName(instance)) — two hops, never through AppHost. This keeps storage traffic off the code DO's single-threaded input gate. AppHost pushes trusted caps down to them (e.g. setLimits → setSqlLimits); the stubs still validate/quota every call.

09Security model & invariants

The sandbox (invariant #1)

  • App isolate env = { SYSTEM: broker } — nothing else.
  • globalOutbound: null — zero network egress; the only way out is mediated requestFetch to allowlisted hosts.
  • Per-run limits.cpuMs / subRequests bound a runaway run.
  • All capabilities are scoped by props.instance (room id).

The trust boundary

  • Worker + all DOs + capability stubs = trusted framework code.
  • Only the Dynamic Worker is untrusted.
  • The AI assistant is trusted host code; it edits app code only through the build-gated setFiles.
  • New powers = new broker methods, never a raw binding to the app (invariant #2).
