01Framework structure — every part & how they connect

One Worker is the front door. It talks to trusted per-room Durable Objects and spawns the untrusted app in a Dynamic Worker. Only three things are guaranteed to exist for any room: the Entry Worker, the AppHost DO, and the Dynamic Worker that runs the code. Everything else is pulled in on demand.

The one idea
Trusted framework code (Worker + DOs + stubs) holds all the power; the untrusted app starts with a single handle, env.SYSTEM, and must ask for everything. That is why almost every box is OPTIONAL — the app pulls a capability in only when it uses it, and a room that uses nothing but fetch runs on just the three ALWAYS pieces.
Why so many separate DOs (and not facets)?
AppData / AppSql / AppScheduler could each have been a facet under AppHost — same isolation, but a facet has no name, so every call would route through AppHost (3 hops) and contend on the code+realtime DO's single input gate. A top-level DO is addressable by name, so the capability stub reaches it directly (2 hops) and keeps AppHost off the storage hot path (limitation #3). Rule of thumb: parent-private sub-store → facet; anything other callers hit directly → top-level DO. See the Durable Objects section for the full argument.

Two rooms are fully isolated
Every DO here is keyed by the room id, so room A can never see room B's code, data, or realtime state. Create a second room and you get a second, independent copy of this whole picture — sharing only the one Entry Worker and the one lobby.
